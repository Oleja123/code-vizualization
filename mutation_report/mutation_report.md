# Мутационное тестирование — отчёт

## Что такое мутационное тестирование

Обычное покрытие кода показывает что тест **прошёл через строку**. Но не говорит — **проверил ли он что строка делает правильно**.

Мутационное тестирование решает эту проблему. Инструмент (PIT) берёт исходный код и создаёт сотни его копий — **мутантов** — в каждой из которых намеренно сломана одна строка. Затем против каждого мутанта запускаются ваши тесты.

- Тест **упал** → мутант **убит** ✅ — тест достаточно сильный
- Тест **прошёл** → мутант **выжил** ❌ — тест не заметил поломку

Чем больше убитых мутантов — тем лучше тесты.

---

## Какие мутации применялись

| Мутатор | Что делает | Пример |
|---|---|---|
| `CONDITIONALS_BOUNDARY` | Сдвигает границу сравнения | `i <= n` → `i < n` |
| `NEGATE_CONDITIONALS` | Инвертирует условие | `if (x > 0)` → `if (x <= 0)` |
| `MATH` | Меняет арифметический оператор | `result * i` → `result / i` |
| `INCREMENTS` | Меняет направление инкремента | `i++` → `i--` |
| `EMPTY_RETURNS` | Возвращает пустое значение | `return list` → `return []` |
| `FALSE_RETURNS` | Возвращает false | `return isValid()` → `return false` |
| `TRUE_RETURNS` | Возвращает true | `return check()` → `return true` |
| `NULL_RETURNS` | Возвращает null | `return object` → `return null` |
| `PRIMITIVE_RETURNS` | Возвращает 0 | `return count` → `return 0` |

### metrics-service — MetricsCalculator

> Сервис подсчёта метрик C-кода. Вычисляет цикломатическую сложность, глубину вложенности, количество вызовов, return и goto по AST-дереву программы.

| Метрика | До доработки | После доработки |
|---|---|---|
| Классов проверено | 1 | 1 |
| Покрытие строк | 96% (164/170) | **100%** (170/170) |
| Убито мутантов | 66% (109/165) | **87%** (144/165) |
| Сила тестов | 71% (109/154) | **88%** (144/164) |

**Что улучшили:** добавлены тесты на граничные случаи которые ранее не покрывались:

- `UnaryExpr`, `AssignmentExpr`, `ArrayAccessExpr` в подсчёте CC и вызовов
- `DoWhileStmt` в счётчиках return и goto
- `LabelStmt` во всех метриках
- Граничные значения LOC (`end == start`, `end < start`)
- Цепочки `&&` и `||` в условиях
- Вложенные вызовы в аргументах `CallExpr`

**21 мутант выжил** — в основном в ветках обработки редких комбинаций вложенных выражений, которые не встречаются в реальном C-коде.

### flowchart-visualizer — FlowchartBuilder + SVGRenderer

> Сервис построения блок-схем. FlowchartBuilder строит граф узлов из AST, SVGRenderer преобразует граф в SVG-разметку.

#### FlowchartBuilder

| Метрика | До доработки | После доработки |
|---|---|---|
| Классов проверено | 2 | 2 |
| Покрытие строк | 82% (245/299) | **86%** (258/299) |
| Убито мутантов | 66% (96/146) | **67%** (98/146) |
| Сила тестов | 77% (96/125) | **74%** (98/133) |

**Что улучшили:** добавлены тесты на:

- Выбор функции по умолчанию (`main` vs первая в списке)
- `getFunctionNames` — порядок и размер списка
- `buildAllFunctions` — все функции строятся корректно
- Структуру `if-else` — наличие `trueBranch`/`falseBranch`
- Связи узлов циклов — `exitNode`, `loopBody`
- Вложенные циклы
- Операторы после цикла

**Почему сила тестов незначительно снизилась:** новые тесты увеличили количество проверяемых мутантов (знаменатель вырос с 125 до 133), убив при этом только 2 новых. Оставшиеся мутанты сосредоточены в сложной внутренней логике `attachEnd` и `linkLeaves` — методах рекурсивного обхода графа, которые крайне сложно проверить без полного рендеринга SVG.

#### SVGRenderer

| Метрика | Значение |
|---|---|
| Покрытие строк | 43% (576/1351) |
| Убито мутантов | 6% (58/975) |
| Сила тестов | 16% (58/364) |

**Почему низкие показатели:** SVGRenderer содержит 1351 строку кода — это генерация SVG-разметки строковыми конкатенациями для каждого типа узла блок-схемы. Покрыть такой класс unit-тестами практически невозможно без визуальной проверки SVG-вывода — это задача для интеграционных или snapshot-тестов. Низкий показатель SVGRenderer не означает слабость тестовой базы, а отражает природу класса.

## Как интерпретировать результаты

**Mutation Coverage (убито мутантов)** — основная метрика. Показывает какой процент намеренных поломок был обнаружен тестами. 87% означает что из 165 сломанных версий программы тесты поймали 144.

**Test Strength (сила тестов)** — качество покрытых тестами строк. Считается только по тем мутантам, которые вообще покрыты тестами. 88% означает что среди покрытых мутантов тесты убивают 88% из них.

**Line Coverage (покрытие строк)** — классическая метрика. Показывает через какой процент строк прошли тесты. 100% покрытие строк при 87% убитых мутантов означает что все строки выполнялись, но не все граничные случаи проверялись.

## Инструменты

- **PIT** 1.17.0 — движок мутационного тестирования для Java
- **pitest-junit5-plugin** 1.2.2 — поддержка JUnit 5
- **Maven** — сборка и запуск: `mvn test-compile pitest:mutationCoverage`
- Отчёты: `target/pit-reports/index.html` в каждом сервисе
